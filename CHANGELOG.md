- [2.7.20] - 2026-06-30 — fix(telemetry): a data-less SubagentStop re-fire no longer clobbers a good subagent recording. Found by live-testing the new statusline segment: spawning an agent recorded `general-purpose / 385K tok / $0.49`, but seconds later the row had reverted to `agent / $0` — a *later* re-fire of the SAME agent arrived after its transcript had been cleaned up, so it computed `$0` with the `"agent"` fallback name, and the v2.7.13 upsert (which always wrote the latest firing) overwrote the richer earlier row. - [2.7.17] - 2026-06-30 — fix(telemetry): budget-cap's incremental transcript offset was broken — re-summed the whole transcript and inflated cost ~3.8x. Surfaced (again) by real `/sc-status` data: the stored `main_total` read **$4,916 / 19,718 turns** while the actual transcript held only **5,138 usage-messages** worth ~**$1,288**. Root cause: v2.7.15 advanced the byte offset with `new_offset += len(line.encode())` and used text-mode `f.seek(offset)` — but Python text-mode `seek()` only accepts opaque `f.tell()` cookies, not hand-counted byte counts, so the offset drifted, overshot the real file size (51.7 MB stored vs a 33 MB file), tripped the `offset > size` reset, and re-summed the entire transcript on the next call → runaway. Fix: read the transcript in **binary mode** with `readline()` + real `f.tell()` byte positions that `seek()` honors (also correct for non-ASCII, which text-mode byte-counting mangled); and when the offset is past EOF (rotation or a legacy-corrupt offset) reset `main_total` and the main `turn_count` too, so it re-derives cleanly instead of stacking onto an inflated value — meaning a deploy of this version self-heals the corrupted `.session-cost`. Verified against the live 33 MB transcript: fresh-derive = **$1,292.72 / 5,148 turns** (matches an independent per-message sum), and a repeat call on the unchanged file adds exactly $0. 1 new test (offset-exactness across repeated calls, with a non-ASCII line). No hook-count change. 1169 tests passing.
