## Performance Issues

### 1. No hook-level caching (biggest win available)

[Multi-agent observability project](https://github.com/disler/claude-code-hooks-multi-agent-observability) benchmarks:
- **Without cache:** 50â€“100ms per transcript parse, 5â€“10s I/O per minute
- **With 60s TTL cache:** ~1ms read, ~99% hit rate â†’ **50â€“100x improvement**

Supercharger has zero hook-level caching. Every `tool-history-tracker`, `repetition-detector`, `failure-tracker`, `cache-health` read from disk on every call.

**Hooks that should cache:**

| Hook | Cache key | TTL | Win |
|---|---|---|---|
| `tool-history-tracker` read path | `session_id` | 60s | Avoid re-parsing history on every PreToolUse |
| `detect-stack` | `cwd` | Session | Don't re-run git/package detection |
| `session-memory-inject` | `session_id` | 60s | Same memory across multiple PostToolUse calls |
| `config-scan` | `project_dir` | Session | CLAUDE.md scan result stable |
| `agent-router` | `prompt hash` | 30s | Same prompt â†’ same classification |

### 2. Process fork overhead

PostToolUse has 10 async + 4 sync hooks = 14 hook invocations per tool call. Async hooks each spawn a subshell via Claude's hook runner. 10 subshells per tool call adds latency.

**Fix:** One dispatcher script that handles multiple async concerns in a single process, reading a shared state file.

### 3. No timeouts on hooks

[sdk](https://github.com/gowaylee/cchooks) best practice: default 30s timeout, <100ms for fast safety checks.

`safety.sh` has no timeout configured. A regex match should complete in <50ms but has no guard.

### 4. UserPromptSubmit hook count

13 hooks on UserPromptSubmit â€” every prompt starts Claude with 13 script invocations before it even sees the prompt:
- `lesson-recall.sh` (sync)
- `agent-router.sh` (sync)
- `economy-reinforce.sh` (sync)
- `scope-guard.sh contract` (sync)
- `prompt-validator.sh` (sync)
- `reentry-detector.sh` (sync)
- `adaptive-economy.sh` (sync)
- `learn-from-prompts.sh` (async)
- `context-advisor.sh` (async)
- `rate-limit-advisor.sh` (async)

Many of these could be consolidated into a single `prompt-preprocessor.sh`.

### Current coverage vs alternatives

| Guardrail | Supercharger | claude-guard | agent-guardrails-template |
|---|---|---|---|
| Destructive Bash | Yes | Yes | Yes |
| Credential files (.env) | Yes | Partial | Yes |

| Prompt injection (MCP) | Yes | No | No |
| Shell wrapper hiding | Yes | Yes | No |
| Credential scanning output | Yes | No | No |
| Package manager enforcement | Yes | No | No |
| Git history rewriting | Partial | No | No |
| **Path traversal** | No | No | No |
| **Symlink attacks** | No | No | No |
| **Git hooks modification** | No | No | No |
| **GitHub Actions injection** | Partial | No | No |
| **Build artifact injection** | No | No | No |

### Missing security features

1. **Path traversal in Write/Edit paths**
   - Pattern: `../../../etc/passwd`, `%2e%2e%2f` encoding
   - Should check normalized path before allowing write
   - Reference: [claude-code-hooks-mastery](https://github.com/disler/claude-code-hooks-mastery) docs recommend: "blocking path traversal by checking for `..` in file paths"

2. **Symlink attack detection**
   - Pattern: write through a symlink that points outside the project
   - Check: if target is a symlink, resolve and check if resolved path is under project root

3. **Git hooks modification detection**
   - Pattern: Claude editing `.git/hooks/` files
   - Should block by default in safe mode

4. **Build artifact injection**
   - Pattern: modifying `node_modules/.bin/`, `__pycache__/`, `.next/`
   - Can be used for persistence/backdoor

## Token Economy Analysis

### What's good

- Stack standards injection at session start (tier-scaled)
- Economy tier enforcement with `economy-reinforce.sh`
- MCP profile context management
- Session cost tracking + budget caps

### What's missing

1. **Session context injection** â€” [claude-hooks-sdk pattern](https://github.com/hgeldenhuys/claude-hooks-sdk): inject session ID + git branch + time on every UserPromptSubmit
   - Saves 50â€“100 tokens per request by avoiding tool calls for this info
   - Supercharger has this partially via `session-memory-inject` but not on every prompt

2. **Hook output suppression** â€” `suppressOutput: true` in hook JSON prevents noise from entering context
   - Currently only used in `confidence-gate.sh`
   - Most hooks print advisory text that enters context unnecessarily

3. **Prompt cache awareness** â€” stack standards reinject on every SessionStart regardless of cache hit
   - Could be gated on cache hit rate (if cache is fresh, skip reinjection)

4. **Adaptive economy at context % thresholds** â€” `adaptive-economy.sh` exists but could be more aggressive
   - Current thresholds may be too late to prevent overspend
