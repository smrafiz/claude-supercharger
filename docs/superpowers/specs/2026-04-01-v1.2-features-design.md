# v1.2 Features — Design Spec

**Date:** 2026-04-01
**Goal:** Four features: enhanced clarification mode, session summary/handoff, webhook notifications, resume tool
**Scope:** New hooks, tools, lib, rule enhancements — no changes to economy, roles, or MCP systems

## Motivation

Four gaps identified from oh-my-claudecode analysis and user workflow pain points:

1. **Vague prompts waste tokens.** The current clarification mode is 4 bullets. Complex requests need a structured interview before execution.
2. **Context loss across sessions.** When sessions end (compaction, rate limits, user closes terminal), there's no structured way to resume. Users re-explain everything.
3. **No remote awareness.** Desktop notifications only work if you're at your machine. Teams and remote users need Slack/Discord/Telegram alerts.
4. **No resume path.** After a rate limit, users have no tool to retrieve what was happening and generate a paste-ready resume prompt.

## Design

### 1. Enhanced Clarification Mode

#### Lightweight (default)
Active on all prompts. Scan for:
- Vague verbs ("fix it", "make it better") → ask what specifically
- Missing scope ("update the app") → ask which files/functions
- No success criteria → derive a binary pass/fail condition
- Multiple tasks in one request → split and confirm priority

Max 3 questions, then proceed with best understanding.

#### Deep Interview (keyword-triggered)
Activated by: "deep interview" or "interview me"

Score the prompt across 4 dimensions (0-3 each):

| Dimension | What's being assessed |
|-----------|----------------------|
| Scope | Which files/functions? Clear boundaries? |
| Success | What does "done" look like? Binary pass/fail? |
| Constraints | What must NOT change? Dependencies? |
| Context | Why now? What exists? What was tried? |

Behavior by score:
- 9-12: Proceed — prompt is clear enough
- 5-8: Ask about the lowest-scoring dimension, then proceed
- 0-4: Full interview — one question per low dimension, don't proceed until total reaches 8+

After interview, summarize understanding in a numbered list and get explicit approval before executing.

**Implementation:** Replace the Clarification Mode section in `configs/universal/supercharger.md`. Pure rules, no hooks or infrastructure.

### 2. Session Summary & Handoff

#### Auto-summary on compaction (hook)
When compaction triggers, generate a structured summary before the transcript is compressed:

```
## Session Summary — [date]
**Working on:** [one-line description]
**Decisions made:**
- [decision 1]
- [decision 2]
**Files changed:** [list]
**What was tried and failed:** [if any]
**Next steps:** [what remains]
**Resume with:** [paste-ready prompt for next session]
```

Saved to: `~/.claude/supercharger/summaries/YYYY-MM-DD-HHMMSS.md`

#### Manual summary (keyword-triggered)
Activated by: "session summary"

Claude generates the same structured block in the conversation. User can copy it for the next session.

#### Rate limit behavior (rule)
When Claude detects a rate limit or session ending unexpectedly, treat it as a compaction event — generate the session summary immediately before stopping.

**Implementation:**
- Enhance `hooks/compaction-backup.sh` — save raw transcript (existing) + create summaries directory
- Add "Session Summary" section to `supercharger.md` with:
  - The summary format template (so Claude knows the structure)
  - Keyword trigger: "session summary" generates the block on demand
  - Compaction rule: "After context compaction, your first response must be a session summary in the format above. Save it as a code block the user can copy."
  - Rate limit rule: "If you detect you're about to be rate limited, generate the session summary immediately before stopping."
- The hook handles file I/O (transcript backup, creating summaries dir). The rules handle Claude's behavior (generating the structured summary). No hook can generate AI content — that's Claude's job via rules.
- Summaries directory: `~/.claude/supercharger/summaries/`
- Note: The summary .md files are created by Claude (via rules telling it to write the summary), not by the hook. The hook ensures the directory exists and the transcript is preserved.

### 3. Webhook Notifications

#### Setup
Post-install tool: `bash tools/webhook-setup.sh`

Interactive flow:
1. Select platform: Slack / Discord / Telegram / Custom URL
2. Paste webhook URL (or bot token + chat ID for Telegram)
3. Send test notification
4. Save config

Config stored at: `~/.claude/supercharger/webhook.json`

Standard format:
```json
{
  "platform": "slack",
  "url": "https://hooks.slack.com/services/...",
  "enabled": true
}
```

Telegram format:
```json
{
  "platform": "telegram",
  "bot_token": "...",
  "chat_id": "...",
  "enabled": true
}
```

#### Hook behavior
Enhances existing `notify.sh`:
- Desktop notification (existing behavior — always)
- If webhook.json exists and enabled, also send to webhook

Payload (JSON):
```json
{
  "text": "Claude Code needs your input",
  "project": "[project directory name]",
  "timestamp": "2026-04-01T12:00:00Z"
}
```

Platform-specific formatting:
- Slack: POST JSON with `{"text": "..."}`
- Discord: POST JSON with `{"content": "..."}`
- Telegram: POST to /sendMessage with `{chat_id, text}`
- Custom: POST raw JSON payload to URL

#### Session completion notification
New hook event: Stop (fires when Claude session ends). Sends a summary notification:

`"Claude Code session complete — [project name]"`

Only fires if webhook is configured.

#### CLI commands
```bash
bash tools/webhook-setup.sh              # interactive setup
bash tools/webhook-setup.sh disable      # disable webhook
bash tools/webhook-setup.sh enable       # re-enable
bash tools/webhook-setup.sh test         # send test message
bash tools/webhook-setup.sh status       # show current config
```

**Implementation:**
- New: `tools/webhook-setup.sh` — interactive setup, test, enable/disable
- New: `lib/webhook.sh` — shared functions for sending webhooks (used by hooks)
- Modify: `hooks/notify.sh` — add webhook call after desktop notification
- New: `hooks/session-complete.sh` — fires on Stop event, sends completion webhook
- Modify: `lib/hooks.sh` — register session-complete hook when webhook is configured
- Config: `~/.claude/supercharger/webhook.json`
- Zero dependencies: uses `curl` for all HTTP calls

### 4. Resume Tool

#### CLI
```bash
bash tools/resume.sh
```

Reads the most recent session summary from `~/.claude/supercharger/summaries/` and outputs it formatted for pasting into a new Claude Code session.

Behavior:
- Finds latest .md file in summaries/
- Extracts the "Resume with:" section
- Prints it to stdout
- If `pbcopy` (macOS) or `xclip` (Linux) available, also copies to clipboard
- If no summaries found, prints: "No session summaries found. Say 'session summary' in Claude Code to generate one."

#### Additional commands
```bash
bash tools/resume.sh --list                      # list past summaries
bash tools/resume.sh --show 2026-04-01-143022.md  # view specific summary
```

**Implementation:**
- New: `tools/resume.sh` — ~50 lines, reads summaries, extracts resume block, clipboard copy
- Depends on: session summary feature (Section 2) producing the summary files
