# v1.7 Implementation Plan — Automation & Learning

## Overview

v1.7 makes Supercharger adaptive: it learns from past sessions, reduces permission friction, auto-adjusts economy tiers, and monitors subagent activity. The theme is **automation that gets smarter the more you use it**.

**Prerequisite**: v1.6.0 must be shipped first. This plan assumes v1.6 delivered ~170 tests, ~12 hooks (full mode), and all 8 Intelligence Layer features.

## Feature 1: Learn from Sessions

### What it does
Analyzes past Claude Code conversation transcripts to suggest improvements to CLAUDE.md and rule files. Finds patterns in what the user corrects, what Claude gets wrong repeatedly, and what instructions are missing.

*Deferred from v1.6 roadmap. Inspired by: review-claudemd skill from claude-code-tips (ykdojo)*

### How it works
1. Reads recent session JSONL files from `~/.claude/projects/`
2. Extracts user corrections (messages after Claude errors), repeated patterns, and frequently used commands
3. Compares against current CLAUDE.md and rule files
4. Outputs suggestions grouped by category:
   - **Missing rules**: Patterns Claude violates that have no corresponding instruction
   - **Weak rules**: Instructions that exist but Claude still violates (need reinforcement)
   - **Outdated rules**: Instructions that reference deprecated patterns
   - **Suggested additions**: New conventions detected from user behavior

### Implementation
1. Create `tools/learn-from-sessions.sh`
   - Embedded python3 script
   - Scans `~/.claude/projects/*/` for JSONL session files from last 7 days (configurable with `--days N`)
   - Parses each JSONL file for message pairs (user correction after assistant error)
   - Pattern detection:
     - User says "no", "wrong", "actually", "I said", "don't" → marks previous assistant message as error
     - User provides the same instruction 3+ times across sessions → missing rule
     - User edits a file Claude just edited → possible mistake correction
   - Compares detected patterns against `~/.claude/CLAUDE.md` and `~/.claude/rules/*.md` content
   - Outputs structured suggestions with confidence scores
   - `--apply` flag to automatically append high-confidence suggestions to CLAUDE.md (with backup)
   - `--dry-run` (default) shows suggestions without modifying anything

2. Add to `tools/claude-check.sh`
   - New section: "Session Learning" — shows how many sessions analyzed, top suggestion

3. Tests:
   - Create mock JSONL sessions with known correction patterns
   - Verify pattern detection works
   - Verify suggestions are generated
   - Verify `--apply` creates backup and appends

### Stdin/Output
This is a standalone tool, not a hook. No stdin JSON needed.

```bash
# Usage
bash tools/learn-from-sessions.sh              # Dry run, last 7 days
bash tools/learn-from-sessions.sh --days 30    # Last 30 days
bash tools/learn-from-sessions.sh --apply      # Auto-apply high-confidence suggestions
```

### Output format
```
📚 Session Learning Report
━━━━━━━━━━━━━━━━━━━━━━━━━
Sessions analyzed: 23 (last 7 days)
Corrections detected: 8
Patterns found: 3

🔴 Missing Rules (high confidence):
  1. User corrects "don't add comments to code" 4 times
     → Suggest adding to CLAUDE.md: "Never add code comments unless explicitly asked"

🟡 Weak Rules (medium confidence):
  1. Rule exists: "No unrequested refactoring"
     → But Claude still refactors in 3/23 sessions
     → Suggest reinforcing with: "NEVER refactor code unless the user explicitly asks for refactoring"

🟢 Conventions Detected:
  1. User always uses pnpm (detected in 18/23 sessions)
     → Already covered by enforce-pkg-manager hook ✓
```

### JSONL Parsing Details
Each session file has lines like:
```json
{"type": "human", "message": {"content": "fix the login bug"}, "timestamp": "..."}
{"type": "assistant", "message": {"content": "I'll refactor the auth module..."}, "timestamp": "..."}
{"type": "human", "message": {"content": "no, don't refactor, just fix the bug"}, "timestamp": "..."}
```

The parser should:
- Read all lines, build (human, assistant) pairs
- Look for correction markers in human messages that follow assistant messages
- Track the topic/context of the correction
- Aggregate across sessions to find repeated patterns (3+ occurrences = high confidence)

**Caveat**: JSONL format may vary by Claude Code version. Parse defensively — if a line doesn't match expected format, skip it. Never crash on malformed data.

## Feature 2: Smart Command Auto-Approval

### What it does
A `PermissionRequest` hook that auto-approves read-only, non-destructive commands. Reduces permission fatigue without disabling safety.

*Inspired by: Dippy (ldayton) — AST-based safe command auto-approval*

### Why this matters
Claude Code asks permission for every `ls`, `git status`, `cat`, `grep`. This breaks flow state. Dippy solves this with a full bash parser, but it's a compiled Python package. We need a lightweight bash+python3 version that covers the 80% case.

### Safe commands (auto-approve)
```yaml
Read-only filesystem:
  - ls, ll, la, tree, find (without -delete/-exec rm)
  - cat, head, tail, less, wc, file, stat, du, df
  - realpath, dirname, basename, pwd

Read-only git:
  - git status, git log, git diff, git branch, git show
  - git remote -v, git stash list, git tag -l

Read-only tools:
  - grep, rg, ag, ack, fgrep, egrep
  - which, where, type, command -v, hash
  - echo, printf (to stdout only, no redirects)
  - date, uname, whoami, hostname, env, printenv
  - node -e (with read-only patterns), python3 -c (with read-only patterns)

Package inspection:
  - npm list, npm ls, npm outdated, npm view, npm info
  - pip list, pip show, pip freeze
  - cargo tree, cargo metadata
  - go list, go version

Process inspection:
  - ps, top (non-interactive), pgrep, lsof
```

### Dangerous patterns (never auto-approve, even in safe commands)
```yaml
Always block auto-approval if command contains:
  - Redirects to files: >, >>, 2>, &>, tee
  - Pipes to destructive commands: | rm, | dd, | xargs rm
  - Command substitution with writes: $(rm ...), `rm ...`
  - Semicolons/chains with writes: ls; rm, git status && rm
  - Environment modification: export, unset, source, eval
  - Network with writes: curl > file, wget
  - sudo anything
```

### Implementation
1. Create `hooks/smart-approve.sh`
   - Event: `PermissionRequest` | Matcher: (none)
   - Reads stdin JSON: `{ "tool_name": "Bash", "tool_input": { "command": "ls -la" } }`
   - Python3 script classifies the command:
     a. Split on `&&`, `||`, `;`, `|` to get individual commands
     b. For each segment, extract the base command (first word after env vars)
     c. Check base command against whitelist
     d. Check full command for dangerous patterns (redirects, substitutions)
     e. If ALL segments are safe AND no dangerous patterns → approve
     f. Otherwise → pass through (let normal permission flow handle it)
   - Output for auto-approve: `{ "decision": "allow" }`
   - Output for pass-through: exit 0 with no output (default behavior)

2. Add to `get_hooks_for_mode()` in **standard+** mode
   - `hooks+=("PermissionRequest||${hooks_dir}/smart-approve.sh")`

3. **Graceful degradation**: If Claude Code doesn't support `PermissionRequest` event, the hook is registered but never fires. No errors. Test by checking if the hook ever receives input.

4. Logging: When a command is auto-approved, append to audit trail (if audit-trail hook is active):
   ```json
   {"type": "auto-approve", "command": "git status", "timestamp": "...", "reason": "read-only git"}
   ```

### Stdin JSON for PermissionRequest
```json
{
  "tool_name": "Bash",
  "tool_input": {
    "command": "git status && git log --oneline -5"
  },
  "tool_use_id": "abc123",
  "session_id": "sess_xyz",
  "cwd": "/Users/me/project"
}
```

### Output format
```json
{
  "decision": "allow"
}
```
Or for non-Bash tools or unsafe commands: exit 0 with no output (pass through to normal flow).

### Safety guarantees
- **Only Bash tool**: Only evaluates `tool_name === "Bash"`. All other tools pass through.
- **Whitelist, not blacklist**: Commands must match the safe list. Unknown commands are NOT auto-approved.
- **Chain analysis**: `git status && rm -rf /` is NOT approved because `rm` is not on the safe list.
- **No regex on arguments**: We check base command names, not argument patterns. `ls -la /etc/passwd` is fine (ls is safe). But `find . -exec rm {} \;` is NOT fine (the full command contains `rm`).
- **Redirect detection**: Any `>`, `>>`, `2>`, `&>`, `tee` in the command → not auto-approved.

- MODIFY: `lib/hooks.sh` (add hook to standard+ mode, update count)
- MODIFY: `tests/test-install.sh` (update hook count)
- MODIFY: `tests/test-hooks.sh` (add smart-approve tests)
- MODIFY: `tools/claude-check.sh` (add to hook check loop)

## Feature 4: Subagent Monitor

### What it does
`SubagentStart` and `SubagentStop` hooks that track subagent (Task tool) activity, log their work, and optionally enforce limits.

### Why
Subagents can consume significant tokens and run for extended periods. Without monitoring, users have no visibility into what subagents are doing or how much they cost. This hook provides:
- Logging of all subagent activity to the audit trail
- Optional limit on concurrent subagents (configurable, default: no limit)
- Timing information (start/stop pairs)

### Implementation
1. Create `hooks/subagent-monitor.sh`
   - Two modes based on which event fires:
     - `SubagentStart`: Log start, check concurrent limit, optionally block
     - `SubagentStop`: Log stop, calculate duration, report

   - For SubagentStart stdin:
     ```json
     {
       "subagent_id": "sub_123",
       "task": "Research the best testing framework for React",
       "session_id": "sess_abc",
       "cwd": "/Users/me/project"
     }
     ```

   - For SubagentStop stdin:
     ```json
     {
       "subagent_id": "sub_123",
       "session_id": "sess_abc",
       "stop_hook_active": true
     }
     ```

   - Python3 script:
     a. Maintain state in `~/.claude/supercharger/audit/subagents-active.json` (list of active subagent IDs with start times)
     b. On start: add to active list, log to audit trail, check count
     c. On stop: remove from active list, log duration to audit trail
     d. If concurrent limit exceeded (configurable via env var `SUPERCHARGER_MAX_SUBAGENTS`), output block:
        ```json
        { "decision": "deny", "message": "Supercharger: max subagent limit (3) reached. Wait for current subagents to finish." }
        ```

2. Register TWO hooks in `get_hooks_for_mode()` for **full** mode:
   ```bash
   hooks+=("SubagentStart||${hooks_dir}/subagent-monitor.sh")
   hooks+=("SubagentStop||${hooks_dir}/subagent-monitor.sh")
   ```
   The script detects which event fired by checking for `subagent_id` + presence/absence of `task` field.

3. **Graceful degradation**: If Claude Code doesn't support these events, hooks never fire.

### Audit trail entries
```json

{"type": "subagent-stop", "subagent_id": "sub_123", "duration_seconds": 45, "timestamp": "2026-04-02T10:30:45Z"}
```

## Feature 8: Hook Pipeline Composer

### What it does
A tool that lets users create custom hook pipelines — chains of existing hooks that run in sequence for a single event.

### Problem
Currently, hooks are flat: each event can have multiple independent hooks, but they can't share state. A user might want:
- UserPromptSubmit: validate → rewrite → inject context (3 hooks in sequence, output of one feeds the next)
- PostToolUse: scan for injection → compact traces → log to audit (3 hooks, sharing the same tool output)

Claude Code already supports multiple hooks per event, and they run in order. But they don't share state — each gets the original input.

### Solution
A composer tool that generates a single "pipeline" hook script that chains multiple hooks:

```bash
bash tools/hook-compose.sh \
  --event UserPromptSubmit \
  --chain "prompt-validator,prompt-rewriter" \
  --name "validate-and-rewrite"
```

This generates `hooks/validate-and-rewrite.sh` that:
1. Runs prompt-validator.sh with the original input
2. If prompt-validator outputs a modified prompt, feeds that to prompt-rewriter.sh
3. Returns the final output

### Implementation
1. Create `tools/hook-compose.sh`
   - Arguments: `--event`, `--chain` (comma-separated hook names), `--name`
   - Validates that all hooks exist in `hooks/` directory
   - Generates a new hook script that chains them
   - The generated script:
     a. Reads stdin JSON
     b. Pipes through first hook
     c. If first hook outputs modified input, uses that for second hook
     d. If first hook blocks (exit 2), the pipeline stops and blocks
     e. Returns the final hook's output
   - Registers the composed hook in place of the individual hooks

2. Template for generated pipeline hook:
   ```bash
   #!/usr/bin/env bash
   # Generated by Supercharger hook-compose
   # Pipeline: prompt-validator → prompt-rewriter
   # Event: UserPromptSubmit
   set -euo pipefail

   INPUT=$(cat)
   HOOKS_DIR="$(dirname "$0")"

   # Stage 1: prompt-validator
   RESULT=$(echo "$INPUT" | bash "$HOOKS_DIR/prompt-validator.sh" 2>/dev/null) || {
     EXIT_CODE=$?
     [ $EXIT_CODE -eq 2 ] && exit 2  # Blocked
     # Error in hook, pass through
   }

   # If stage 1 returned modified input, use it
   if [ -n "$RESULT" ]; then
     # Check if result has "input" field (modified prompt)
     MODIFIED=$(echo "$RESULT" | python3 -c "
   import json, sys
   try:
       d = json.load(sys.stdin)
       if 'input' in d:
           # Merge modified input back into original
           import json as j
           orig = j.loads('''$INPUT''')
           orig['input'] = d['input']
           print(j.dumps(orig))
       else:
           print('')
   except: print('')
   " 2>/dev/null)
     [ -n "$MODIFIED" ] && INPUT="$MODIFIED"
   fi

   # Stage 2: prompt-rewriter
   echo "$INPUT" | bash "$HOOKS_DIR/prompt-rewriter.sh"
   ```

3. Add `--decompose` flag to undo a pipeline:
   ```bash
   bash tools/hook-compose.sh --decompose "validate-and-rewrite"
   ```
   This deletes the pipeline script and re-registers the individual hooks.
